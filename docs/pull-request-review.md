# Pull Request 검토 체크리스트

Pull Request(PR)를 병합하기 전에 변경의 목적, 정확성, 검증 범위와 운영 영향을
확인하기 위한 체크리스트입니다. 모든 항목을 기계적으로 적용하기보다 변경의
위험도와 영향 범위에 맞춰 사용합니다.

## 작성자 확인

- [ ] PR 제목과 설명만으로 변경 목적을 이해할 수 있다.
- [ ] 요구사항과 무관한 변경이 포함되지 않았다.
- [ ] 큰 변경은 검토 가능한 단위로 나누었다.
- [ ] 임시 코드, 디버그 출력과 사용하지 않는 파일을 제거했다.
- [ ] 사용자에게 보이는 동작 변경과 호환성 영향을 설명했다.
- [ ] 관련 이슈, 설계 문서 또는 근거 자료를 연결했다.

## 동작과 정확성

- [ ] 정상 흐름과 주요 예외 흐름이 요구사항에 맞는다.
- [ ] 경계값, 빈 입력, 중복 입력과 잘못된 입력을 처리한다.
- [ ] 오류를 숨기거나 성공으로 오인하게 만드는 처리가 없다.
- [ ] 재시도 또는 재실행 시 데이터가 중복되거나 손상되지 않는다.
- [ ] 시간대, 정렬 순서, 숫자 정밀도와 문자 인코딩을 확인했다.
- [ ] 동시 실행 시 공유 상태와 경합 조건(race condition)을 검토했다.

## 인터페이스와 데이터

- [ ] 공개 API 또는 함수 계약의 변경이 의도적이며 문서화되었다.
- [ ] 기존 호출자와 저장 데이터에 대한 하위 호환성을 검토했다.
- [ ] 스키마 변경에 적용과 되돌리기(rollback) 방법이 있다.
- [ ] 개인정보와 비밀값이 로그, 예외 또는 응답에 노출되지 않는다.
- [ ] 입력 검증과 권한 검사가 신뢰 경계(trust boundary)에 배치되었다.

## 보안과 의존성

- [ ] 사용자 입력이 SQL, 셸, 템플릿 등에 안전하게 전달된다.
- [ ] 인증(authentication)과 인가(authorization)를 혼동하지 않았다.
- [ ] 새 의존성이 꼭 필요하며 버전과 라이선스를 확인했다.
- [ ] 토큰, 비밀번호와 인증서가 코드나 테스트 데이터에 포함되지 않았다.
- [ ] 민감한 오류 상세정보를 외부 사용자에게 반환하지 않는다.

## 성능과 운영

- [ ] 반복문 내부의 네트워크 또는 데이터베이스 호출을 점검했다.
- [ ] 데이터 증가에 따른 시간·메모리 복잡도를 검토했다.
- [ ] 타임아웃, 재시도와 실패 복구 방식이 명확하다.
- [ ] 필요한 로그, 메트릭과 추적 정보(observability)를 제공한다.
- [ ] 배포 중 구버전과 신버전이 함께 동작하는 상황을 고려했다.

## 테스트와 문서

- [ ] 변경된 핵심 동작을 자동 테스트가 검증한다.
- [ ] 회귀 가능성이 있는 기존 테스트를 실행했다.
- [ ] 테스트가 구현 세부사항보다 외부 동작을 검증한다.
- [ ] 실패를 재현하거나 수동으로 검증하는 명령을 기록했다.
- [ ] 사용법, 설정값 또는 운영 절차 변경을 문서에 반영했다.

## 병합 판단

- [ ] 필수 검사와 Continuous Integration(CI)이 통과했다.
- [ ] 해결되지 않은 검토 의견의 처리 여부가 명확하다.
- [ ] 남은 위험과 후속 작업이 이슈로 추적된다.
- [ ] 장애 발생 시 되돌리거나 기능을 비활성화할 수 있다.

## 검토 의견 작성 기준

검토 의견에는 심각도와 요청 의도를 함께 표시하면 의사소통 비용을 줄일 수
있습니다.

| 표시 | 의미 |
| --- | --- |
| `blocking` | 병합 전에 반드시 해결해야 하는 결함 또는 위험 |
| `suggestion` | 품질 개선 제안이며 작성자와 대안을 논의할 항목 |
| `question` | 의도나 근거를 확인하기 위한 질문 |
| `nit` | 동작에 영향이 없는 사소한 표현 또는 형식 의견 |
